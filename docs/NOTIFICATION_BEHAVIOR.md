# 通知和播报行为详解

## 概述

本文档详细说明应用在不同状态下的通知和语音播报行为。

## 通知系统工作原理

### 系统级通知调度

提醒助手使用 `expo-notifications` 的 **DAILY 调度**：

```javascript
trigger: {
  type: SchedulableTriggerInputTypes.DAILY,
  hour: 9,
  minute: 30,
}
```

这意味着：
- ✅ 通知由 **Android/iOS 系统**管理，不依赖应用运行
- ✅ 应用完全退出后，系统仍会在指定时间发送通知
- ✅ 通知在开机后自动恢复（RECEIVE_BOOT_COMPLETED 权限）

## 应用不同状态下的行为

### 1. 应用在前台运行

```
时间到达 → 系统发送通知
         ↓
    触发 addNotificationReceivedListener
         ↓
    延迟 0.5 秒
         ↓
    自动语音播报 ✅
```

**体验：** 最佳，自动播报

### 2. 应用在后台运行（未完全退出）

```
时间到达 → 系统发送通知
         ↓
    显示通知栏通知 ✅
         ↓
    应用在后台睡眠 ❌
         ↓
    不触发 addNotificationReceivedListener
         ↓
    不会自动播报
    
用户操作：
  点击通知 → 应用前台化
           ↓
      触发 addNotificationResponseReceivedListener
           ↓
      延迟 1 秒
           ↓
      语音播报 ✅
```

**体验：** 需要点击通知

### 3. 应用完全退出（Kill）

```
时间到达 → 系统发送通知
         ↓
    显示通知栏通知 ✅
    显示锁屏通知 ✅
    振动 + 声音 ✅
         ↓
    应用未运行 ❌
         ↓
    不会自动播报
    
用户操作方式一：
  点击通知主体 → 启动应用
               ↓
          触发 addNotificationResponseReceivedListener
               ↓
          延迟 1 秒
               ↓
          语音播报 ✅

用户操作方式二：
  点击"🔊 播报"按钮 → 启动应用
                    ↓
               触发 addNotificationResponseReceivedListener
               (actionIdentifier = 'speak')
                    ↓
               延迟 0.5 秒
                    ↓
               语音播报 ✅

用户操作方式三：
  点击"✓ 完成"按钮 → 关闭通知
                    ↓
               不启动应用
                    ↓
               不播报
```

**体验：** 需要点击通知或按钮

### 4. 锁屏状态（应用已退出）

```
时间到达 → 系统发送通知
         ↓
    锁屏显示通知 ✅
    (lockscreenVisibility: PUBLIC)
         ↓
    振动 + 灯光 ✅
         ↓
    应用未运行 ❌
         ↓
    不会自动播报
    
用户操作（无需解锁）：
  在锁屏上点击"🔊 播报" → 解锁并启动应用
                        ↓
                   触发监听器
                        ↓
                   语音播报 ✅
```

**体验：** 一键操作（锁屏状态下）

## 技术限制说明

### 为什么应用退出后不能自动播报？

#### Android 系统限制

1. **后台执行限制**（Android 8.0+）
   - 应用退出后，所有 JavaScript 代码停止执行
   - 不允许后台服务随意播放音频
   - 需要前台服务（会显示持久通知）

2. **电池优化**
   - 系统会主动杀死后台进程
   - 即使使用 foreground service，也可能被限制

3. **Expo Managed Workflow 限制**
   - 无法直接添加原生代码
   - 无法访问 Android AlarmManager + 全屏 Activity
   - 无法实现类似闹钟应用的行为

#### expo-speech 限制

```javascript
// expo-speech 要求
await Speech.speak(text);
// ↑ 需要应用在前台运行
// ↑ JavaScript 引擎必须活跃
// ↑ 无法在后台/应用退出时工作
```

### 其他应用如何做到的？

**闹钟应用** (如 Google Clock):
```
使用原生 Android AlarmManager
        ↓
触发全屏 Activity
        ↓
播放音频（原生媒体播放器）
```

**微信/钉钉语音通话**:
```
使用 FCM/推送服务
        ↓
启动前台服务
        ↓
显示持久通知 + 播放铃声
```

## 解决方案对比

### 方案 A：当前实现（推荐）✅

**优点：**
- ✅ 保持 Expo Managed 简洁性
- ✅ 通知100%可靠
- ✅ 一键触发播报
- ✅ 无需额外配置

**缺点：**
- ❌ 需要用户点击

**用户操作：** 点击通知或"🔊 播报"按钮

---

### 方案 B：使用前台服务

**实现：**
```javascript
// 应用启动时开启前台服务
await Notifications.setNotificationHandler({
  handleNotification: async () => ({
    shouldShowAlert: true,
    shouldPlaySound: true,
  }),
});

// 保持应用在后台活跃
```

**优点：**
- ✅ 可以在应用"退出"后播报

**缺点：**
- ❌ 会显示**持久通知**（无法关闭）
- ❌ 消耗更多电池
- ❌ 用户体验不佳
- ❌ 违反 Android 最佳实践

---

### 方案 C：切换到 Bare Workflow + 原生模块

**实现：**
```bash
expo eject
# 添加原生 Android TTS 服务
# 使用 AlarmManager + BroadcastReceiver
```

**优点：**
- ✅ 完全自动播报
- ✅ 类似闹钟应用体验

**缺点：**
- ❌ 失去 Expo 托管的便利性
- ❌ 需要维护原生代码
- ❌ OTA 更新受限
- ❌ 开发复杂度高

---

### 方案 D：使用预录音频

**实现：**
```javascript
import { Audio } from 'expo-av';

// 预先录制所有提醒的音频文件
const sound = new Audio.Sound();
await sound.loadAsync(require('./assets/reminder-9-30.mp3'));
await sound.playAsync();
```

**优点：**
- ✅ 可以通过通知触发音频
- ✅ 不需要原生代码

**缺点：**
- ❌ 需要录制大量音频文件
- ❌ 不够灵活（无法动态变化）
- ❌ 增加应用体积
- ❌ 多语言支持困难

## 推荐的使用方式

### 确保通知可靠送达

1. **允许应用通知权限**
   ```
   设置 → 应用 → 提醒助手 → 通知 → 允许
   ```

2. **关闭电池优化**（部分手机需要）
   ```
   设置 → 电池 → 电池优化 → 提醒助手 → 不优化
   ```

3. **允许后台运行**（部分手机需要）
   ```
   设置 → 应用 → 提醒助手 → 电池 → 允许后台活动
   ```

4. **锁屏通知显示**
   ```
   设置 → 锁屏 → 显示所有通知内容
   ```

### 获得最佳播报体验

**推荐做法：**
- 在训练时间段保持应用在前台或后台运行
- 收到通知时点击"🔊 播报"按钮
- 重要提醒可设置多个时间点

**避免：**
- 完全退出应用后期待自动播报
- 在勿扰模式下期待通知

## 常见问题

**Q: 我完全退出应用后，第二天早上还会有提醒吗？**
A: ✅ 会的！通知是由系统管理的，即使应用退出也会按时发送。

**Q: 通知会自动播报吗？**
A: 应用前台运行时会自动播报，退出后需要点击通知。

**Q: 我能让它像闹钟一样自动响吗？**
A: 目前技术架构不支持。需要切换到 Bare Workflow 并开发原生模块。

**Q: 点击通知后多久会播报？**
A: 点击"🔊 播报"按钮约 0.5 秒，点击通知主体约 1 秒（等待应用完全启动）。

**Q: 如果我在开会，不想让通知显示怎么办？**
A: 可以在应用内禁用该项目，或者使用系统勿扰模式（但会阻止通知）。

## 总结

- ✅ **通知 100% 可靠**（即使应用退出）
- ✅ **播报功能完善**（需要用户触发）
- ⚠️ **无法完全自动**（Android 系统限制）
- 🎯 **当前方案最优**（在 Managed Workflow 下）

如需完全自动播报，建议：
1. 保持应用在后台运行
2. 或接受点击通知的交互方式
3. 或切换到 Bare Workflow（需大量开发工作）
